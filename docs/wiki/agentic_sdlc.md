# Agentic SDLC Wishlist

Frontier LLMs and agentic coding tools such as Claude Code have made great strides in accelerating individual parts of the software development process.  But we're still learning how to best leverage these new capabilities and avoid just moving the bottlenecks around or leaving agents flailing around trying to achieve an assigned task without the proper tools to get the job done effectively.  What I describe here is what I'd like to see from a more fully automated software development lifecycle that respects agent capabilities, the value of human time and contributions, and the power of using the right tools for the job.

## Context Curation

Tools like Claude Code are relatively good at figuring out how to access the information needed to figure out how to perform a task.  But they're sometimes a little too eager to go out and hunt for things themselves, often stumbling across things that sound reasonable to the LLM but pre-dispose it to an approach that the human has already carefully evaluated and dismissed, or missing crucial background information that they assume simply isn't available.  Humans often have a collection of references relevant to the tasks they want completed, but in a poorly organized state; bookmarks, emails, downloaded PDFs, web services, paper documents, ideas never written down, etc.  Especially in the early stages of planning a project or feature, the agent should really nudge the human to fill in any context gaps with existing references or knowledge the human already has before starting to hunt independently.  And the process of loading that information from assorted sources should be as streamlined as possible.

Tools such as LLM-Wiki and derivations of it are pretty good at the mechanics of loading context from various sources, but tend to be more independent than is really appropriate for this stage of the process.  Before running off to the internet and guessing what use to fill gaps in the stated intent, the agent should really work with the human to clarify what's desired and where to look first.  I've seen individual skills that do aspects of this, but haven't experienced a good unified experience for it; I always have to stitch several unrelated skills together to really get an agent to work with me on collecting background information and clarifying intent. (A small `context-curation-check` skill for this is sketched but not yet written for PortaBase — see `agentic-development-support.md`.)

## Progressive Discovery

Most information sources are terrible at guiding agents (or humans) to exactly the information they need.  We end up having to read a reference manual when we really want a howto, end up with a brief howto when we're really trying to figure out *why* something works the way it does, many loosely related topics will be covered in a lengthy poorly-indexed document, etc.  Agents end up needing to read all kinds of crap to find what they're looking for, and once they eventually figure it out, typically just complete the original request and then throw away that hard-won understanding.  Humans tend to ask agents to perform a sequence of related tasks over time, working with the same codebase or in the context of the same organization and business model.  The agent should write up its current understanding of all that relevant context in well-indexed, cross-linked, appropriately-sized chunks of information.  This way the next time that information is needed, it's much easier to find and load into context, and it can be improved over time.  The LLM-Wiki model is a reasonable stab at this, but the original implementation again tends to run off on its own a little too much.  Ideally, the agent should follow a context organization framework to propose updates to the knowledge repository, which the human then reviews and corrects as needed.  I have a reasonably good implementation of this at work, but can't use it for personal projects so I'm looking for something else that can also do this well enough. (PortaBase trialed OpenKnowledge for its `docs/wiki/` in mid-2026 and dropped it — real reliability bugs on top of the same run-off-on-its-own problem; see `openknowledge-bugs.md` and `agentic-development-support.md`. A follow-up sweep of ~150 tools descended from the same LLM-Wiki pattern, including well-funded newcomers, didn't turn up anything better. Still an open search as of this writing.)

## Context Indexing & Search

Agentic coding assistants can use reasonably good search engines on the internet, but typically fall back to grep when dealing with local files.  There's really no excuse for this when there are good and fast full text search engines like Tantivy in Turso, or semantic search capabilities with compact indexes such as LEANN.  There are similarly good open source source code index engines.  When utilizing context in local source code or knowledge bases, the agent should use the best available search technology (and the human should also have reasonably good content search available to them). (This is a live gap for PortaBase right now, not a hypothetical one: with OpenKnowledge dropped, `docs/wiki/` search is back to grep/`Read`. It matters most for `.reference-repos/` — those clones can be far larger than PortaBase's own source — but isn't blocking anything today, so it's parked rather than actively worked.)

## Hierarchical, Linked Specifications

The basic idea of spec-driven development is a good one, but the details really matter.  Some attempts (SpecKit, maybe BMAD) produce extremely detailed specifications of *what* the code is supposed to be, which needs to be reviewed in essentially as much detail as the source code normally would be.  And then the human has to either also review the source code, or hope that it ends up correct without real verification.  Many such systems are also one-way, with no real mechanism for ensuring that the specifications stay current with the source code.  Linked-Intent Development (LID) seems to handle the 2-way linkage between high-level specs, low-level specs, and code better, with discovery of problems with the specs during implementation triggering appropriate spec updates that cascade back to the source code. (Piloted against PortaBase's column-editing subsystem in mid-2026; see `lid-feedback.md` for concrete inconsistencies the pilot found in LID's own docs.)

But what's really valuable are specs that primarily explain *why* key decisions were made.  If the spec just says "do it this way", both agents and humans struggle to make good judgement calls when changes are later needed to that part of the code.  Arguably, the spec doesn't even need to be as detailed (and difficult to review) if it explains which decisions were made for very specific reasons (and by omission, which were relatively arbitrary implementation details that should be safe to change).

## Support for Multi-Repository Development

Not as much of a concern for PortaBase, but many projects have source code split among multiple repositories for pretty good reasons.  They may represent independently deployed services, team ownership boundaries, explicit choices to limit the testing scope needed for any one change, etc.  But most spec-driven development systems start to fall apart beyond one repository.  Ideally each repo should have its own specs, but they can be linked to higher level specs kept in separate repositories and/or a project knowledge base of the type outlined above.  And the agentic development process should be able to smoothly handle figuring out which changes are needed in which repos, and sequence the updates accordingly.

## Multiple Agents with Clearly Defined Roles

Telling one agent to do everything inevitably starts to fall apart.  It becomes biased in defending its original design in the face of mounting evidence that a different design may work better, it becomes convinced that every single flaw it finds during review, no matter how trivial, must be explicitly addressed, etc.  Really, separate agents/subagents are needed for testable design, test implementation, code implementation, different review dimensions, triage of review results, etc.  It's unclear if giving each role a specific persona really helps, but separation of duties definitely does.

## State Machine for Agent Orchestration

Processes like TDD have proven to work very well in software development, but humans have trouble following them reliably.  So do agents.  Based on its training data an LLM will be really inclined to write code before the tests, an agent that's just supposed to make the tests pass will start modifying the tests whenever convenient, agents responsible for fully enumerating all discovered defects will just stop after they think they've found a reasonable number, an agent responsible for ensuring convergence between changes and review findings will declare "mission accomplished" even though significant changes were just made after the last round of review, etc.  There needs to be a state machine orchestrating the entire development process, either running as a deterministic script that launches agentic development tool instances or a main agent that's purely responsible for advancing the state of the state machine, delegating everything else to subagents or collaborator agents.
